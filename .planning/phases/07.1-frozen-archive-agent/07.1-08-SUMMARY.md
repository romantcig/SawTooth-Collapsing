**以单调 request_id 串联请求入口、Archive 召回、上游发送与错误路径，并将逐块 Archive Info 噪音收敛为每请求一次最终结果汇总。**

## Accomplishments

- 新增 `requestMeta` 和 `Server.requestIdx`，从 `HandleMessages` 最早入口分配 process-local 单调 ID，并用固定 `request_id`、`request_session_id` 属性贯穿入口、管线、上游响应和错误日志。
- 将“请求已接收/message_count”拆成“请求进入/original_message_count”和“上游请求发送/forwarded_message_count”，保留 model family，同时不记录 Authorization、body、prompt 或完整 system。
- `SearchAndExpand` 接入 request-scoped meta，逐 block 注入与预算耗尽详情降为 Debug，来源 session 改用 `source_session_id`，显式召回最终只输出一条含完整 `RecallOutcome` 字段的 Info 汇总。
- 新增并发集成测试，让两条请求日志自然交错，再仅按 request_id 分组验证每组均含入口、最终召回和上游发送，且不会跨组混淆当前 session。

## Decisions Made

- 使用 `atomic.Uint64` 满足单进程、低成本、并发唯一的审计需求；ID 只保证当前进程生命周期内单调，不引入持久化或 UUID 复杂度。
- request-scoped 属性由现有 `LogHandler.WithAttrs` 承载，继续共享 handler mutex 保证单行原子性；请求链重建依赖结构化键，而非输出行邻接或 goroutine-local 聚合。
- 无显式召回信号的请求不输出 Archive Info；存在显式尝试时无论命中、预算丢弃或注入，均由 defer 基于最终 `RecallOutcome` 输出恰好一条真实状态汇总。
