## Critical Issues

### CR-01: Debug 请求快照会把 Anthropic API key 明文写入磁盘

**File:** `internal/proxy/forward.go:110-120`

**Issue:** `writeDebugFile` 只对 `Authorization` 做脱敏，其他 header 全部原样写入 debug JSON。Anthropic 客户端通常可通过 `x-api-key` 携带凭证，因此开启 debug 后，API key 会以明文落到 `data_dir/debug/.../*-req.json`。调用链是 `forwardRaw` 的 `r.Header` → `writeDebugFile` → `headersMap` → `os.WriteFile`，属于明确的凭证泄漏。

**Fix:** 对所有认证/会话凭证 header 做大小写不敏感的白名单式脱敏，至少覆盖 `Authorization`、`Proxy-Authorization`、`X-Api-Key`、`Anthropic-Api-Key`、`Cookie`；更稳妥的做法是 debug 文件只记录经过允许的诊断 header，而不是复制全部 header。增加测试，向请求同时注入上述 header，并断言文件内容不含任何 secret。

### CR-02: 未校验的 session header 可穿越 debug 根目录并在任意可写位置创建文件

**Files:** `internal/proxy/session.go:189-194`; `internal/proxy/forward.go:87-100`

**Issue:** `extractSessionID` 直接信任 `X-Claude-Code-Session-Id`，随后 `writeDebugFile` 将它作为 `filepath.Join(dataDir, "debug", sessionID)` 的路径组件。攻击者可发送包含 `..`、斜杠或反斜杠的 session ID，使 `MkdirAll`/`WriteFile` 逃逸 debug 根目录。代理若监听到非完全可信的本机接口，单个 HTTP 请求即可在进程权限允许的位置创建目录和 JSON 文件。

**Fix:** 不要把原始 session ID 用作路径。使用 `sha256(sessionID)` 或经过严格字符集与长度限制的不可逆目录名，并在写入前用 `filepath.Abs`/`filepath.Rel` 验证最终路径仍位于 `<dataDir>/debug` 内。增加 `../`、`..\\`、绝对盘符和 UNC 路径测试。

### CR-03: Frozen boundary hash 忽略大部分消息内容，用户编辑会被静默丢弃

**Files:** `internal/proxy/frozen.go:418-433`; `internal/proxy/proxy.go:391-418`

**Issue:** `stableBoundaryHash` 只哈希 `role + 第一个 text block 的前 200 字节`。对以下真实边界消息，编辑检测完全失效：首个文本块 200 字节之后的修改、第二个及后续文本块的修改、`tool_use`/`tool_result` 内容修改，以及没有 text block 的消息（此时同 role 的所有内容得到同一个 hash）。`HandleMessages` 在 `StripReminders` 后调用 `Frozen.Get`，因此没有必要再用如此弱的“稳定片段”规避 reminder 噪音。一旦弱 hash 误判命中，代码在 `proxy.go:418` 用旧 Frozen prefix 替换原历史，用户修改不会被发送上游，直接违反“不丢信息”和 boundary edit 契约。

**Fix:** 对已经执行 `StripReminders` 的完整 boundary `Message` 做 canonical JSON SHA-256；不要截断到 200 字节，也不要只选第一个 text block。补充三类回归：修改第 201 字节、修改第二个 text block、修改 tool_result content，均必须使 `Frozen.Get` 返回 nil。

### CR-04: Claude 主请求缺少 thinking 时被分类为 unknown，核心代理功能被永久旁路

**Files:** `internal/proxy/session.go:108-121`; `internal/proxy/proxy.go:323-372`

**Issue:** 分类器只有在 `ModelFamily == claude && ThinkingPresent` 时才把 Claude 认作 main；普通 Claude Sonnet/Opus 请求未启用 extended thinking 时会落入 `unknown`。`HandleMessages` 对 unknown 与 subagent 使用同一个早返回路径，因此该请求不会执行 Frozen、Archive recall、Compress/Collapse、SaveArchive 或 Decay。也就是说，标准的无 thinking Claude Code 主会话会完全失去 Sawtooth Proxy 的核心价值。当前 fixture 只覆盖 DeepSeek main 的 thinking present/absent，没有覆盖 Claude main，因此测试矩阵掩盖了这个分支。

**Fix:** `thinking` 只能作为诊断/fallback 信号，不能决定 Claude main 身份。可靠 subagent marker 优先后，普通 Claude 模型应进入 main，或基于真实脱敏样本增加独立的主/子角色 marker。移除“Haiku 模型名即 subagent”的单一判据，至少增加 Claude Sonnet/Opus 有无 thinking、Claude Haiku main/subagent 的集成测试，并断言 main 会调用主管线。

### CR-05: 损坏的持久化 Frozen cutoff 可在主请求路径触发负下标 panic

**Files:** `internal/proxy/frozen.go:367-401`; `internal/proxy/frozen.go:170-208`; `internal/proxy/proxy.go:412-419`

**Issue:** 冷启动加载只验证 `Messages` 非空和 `PrefixHash`，没有验证 `Cutoff`。若 `frozen_state.value` 中 cutoff 为负数，`Frozen.Get` 的 `len(currentMessages) < cutoff` 为 false，boundary 检查也被跳过，随后返回负 cutoff。主请求路径在 `proxy.go:418` 执行 `messages[result.Cutoff:]`，产生 slice bounds panic。cutoff 为 0 虽不 panic，但会把 Frozen prefix 与完整原请求拼接，造成历史重复。SQLite 状态损坏、旧版本异常值或人工修复错误都可触发该路径。

**Fix:** 在 `loadFrozenFromDB` 写入内存前拒绝 `Cutoff <= 0`、负 token 字段、空 boundary hash 等非法状态；`Get` 返回前再次防御性验证 `0 < cutoff && cutoff <= len(currentMessages)`。`HandleMessages` 也应在切片前 fail closed。增加 cutoff `-1`、`0`、大于当前消息数三种冷启动测试，要求无 panic、无重复拼接并丢弃状态。

## Warnings

### WR-01: `cachedTTL` 在并发请求间无同步读写，存在真实 data race

**File:** `internal/proxy/proxy.go:131-143,819-840`

**Issue:** `Server` 被 HTTP server 并发复用，但 `applyCacheControl` 在多个请求中直接比较并写入 `s.cachedTTL`。`Frozen.UpdateTTL` 和 `Sawtooth.SetPauseThreshold` 自身有锁，唯独 compare-and-set 状态没有同步。并发 session 同时检测 ephemeral/1h 时会触发 race；Go string 是多字机器字表示，竞态读写不受语言内存模型保证。现有“并发日志”测试没有进入带 Frozen breakpoint 的该分支，未覆盖此问题。

**Fix:** 给 TTL 状态增加专用 mutex，将比较、更新以及两个下游 setter 作为一个临界区；或者删除缓存字段并幂等调用线程安全 setter。增加两个并发 Frozen 请求分别携带 ephemeral/1h breakpoint 的 race 测试，并用支持 MinGW 的 Windows race 环境执行。

### WR-02: 同一 session 并发 Store/Update 可把旧 Frozen 快照最后写入 SQLite

**Files:** `internal/proxy/frozen.go:90-133`; `internal/proxy/frozen.go:246-291`; `internal/proxy/store.go:292-304`

**Issue:** `Store`/`UpdateMessages` 先在锁内更新内存，解锁后才调用 `persistFn`。同一 session 的两个并发请求可能按“新状态写内存 → 新状态写 DB → 旧请求最后写 DB”的顺序完成，导致内存是新 snapshot、SQLite 却是旧 snapshot；重启后会恢复过期状态。`PersistState` 的 `INSERT OR REPLACE` 没有版本条件，无法拒绝迟到写入。

**Fix:** 为每个 Frozen 状态加入单调 generation，并在持久层使用条件 upsert 拒绝较小 generation；最小修复也可对同一 thread 的持久化顺序串行化。增加可控阻塞的并发测试，强制旧请求最后调用 persist，并验证冷启动仍恢复较新的 cutoff/messages。

### WR-03: Debug 文件名只有毫秒时间戳，同 session 并发请求会互相覆盖

**Files:** `internal/proxy/forward.go:96-100`; `internal/proxy/forward.go:133-140`; `internal/proxy/request_meta.go:8-25`

**Issue:** 文件名是 `{毫秒时间戳}-{direction}.json`，没有已经存在的 `request_id`。同一 session 在同一毫秒内发起两个请求时，两个 `req` 文件路径相同，`os.WriteFile` 会无提示覆盖；响应也可能覆盖。Phase 08 的请求级审计因此在 debug 载荷层丢失证据。

**Fix:** 将 `request_id` 传入 `writeDebugFile`，文件名至少使用 `{timestamp-nanoseconds}-{request_id}-{direction}.json`，并用 `O_CREATE|O_EXCL` 防止静默覆盖。增加固定相同 timestamp、相同 session、不同 request ID 的并发写入测试。

### WR-04: 只读审计从运行中的 WAL 数据库逐文件复制，无法保证快照一致或归因生产文件变化

**File:** `internal/proxy/archive_readonly_audit_test.go:26-42`

**Issue:** 测试依次复制主 DB、WAL、SHM，但没有冻结生产 writer 或使用 SQLite backup/snapshot 机制。代理运行期间发生 checkpoint/写入时，三个文件可能来自不同时间点，审计结果不可靠甚至无法打开。测试最后比较生产指纹也无法区分“审计导致变化”和“正常代理请求导致变化”，会产生错误的安全结论或假失败。

**Fix:** 明确要求并自动验证生产代理处于 quiescent 状态，或使用受支持的一致性快照/backup 流程。若坚持普通文件复制，至少在复制前后重复读取三件套指纹，仅在整个复制窗口完全稳定时接受快照；否则跳过并提示先停止 writer。增加模拟复制期间 WAL 变化的测试。

### WR-05: `CollapseOldMessages` 声称检查两组输入，实际长度不一致会 panic

**File:** `internal/proxy/collapse.go:84-100`

**Issue:** 当 `len(modified) > 10000` 时无条件执行 `original = original[:10000]`；若 original 更短会立即 panic。即使未超过 10000，只要 `cutoffIdx > len(original)`，`original[:cutoffIdx]` 仍会 panic。当前生产调用通常传入等长 slice，但这是导出函数，注释还明确声称“同时检查 modified 和 original”，实际没有建立该契约。

**Fix:** 在任何切片前验证两者长度；以两者安全最小长度限制上限，并在 `cutoffIdx >= len(original)` 时 fail closed（最好返回 error）。增加 original 较短、modified 超 10000/original 不足 10000 的测试。

### WR-06: Phase 08 的 request-scoped 日志没有贯穿 Agent 与 Frozen 管线

**Files:** `internal/proxy/proxy.go:319-320`; `internal/proxy/session.go:174-186`; `internal/proxy/frozen.go:135-140,170-229,296-309`

**Issue:** `requestMeta` 已建立 request-scoped logger，但 `logAgentRequestFeatures` 和 Frozen Store/Get/Invalidate 仍调用全局 `slog`。并发请求发生 Frozen 命中、boundary 失效或 Agent 分类时，这些关键事件没有 `request_id`/`request_session_id`，无法按 Phase 08 声明的方式还原完整请求链。当前并发测试只要求入口、召回汇总和发送三个事件，因此没有捕获缺口。

**Fix:** 让 Agent/Frozen 操作接受 request logger/meta，或返回结构化结果由 `HandleMessages` 统一记录。增加并发测试，要求每个 request ID 的链中包含 agent classification 和 Frozen hit/miss/invalidate 事件，并禁止这些事件以无 request ID 的全局日志出现。

### WR-07: Archive “canonical hash” 既不真正 canonical，也信任调用方提供的未验证 hash

**Files:** `internal/proxy/collapse.go:166-174`; `internal/proxy/store.go:239-266`

**Issue:** hash 直接基于含 `json.RawMessage` 的 `[]Message` 编码；content 内对象的字段顺序仍可影响字节，因此语义相同但键顺序不同的请求会得到不同 hash，继续制造重复 Archive。另一方面，`SaveArchive` 只在 `ContentHash == ""` 时重算；若调用方提供了错误或复用的非空 hash，同 session/range 的不同正文会命中唯一索引并被 `ON CONFLICT DO NOTHING` 静默丢弃。

**Fix:** 对每条 RawMessage 先反序列化为 JSON value，再使用稳定编码生成 canonical bytes；`SaveArchive` 应始终重算并覆盖 hash，或校验传入值与重算值一致后才写入。增加空白/对象键顺序变化仍幂等，以及伪造相同 hash 的不同正文不能被静默丢弃的测试。
