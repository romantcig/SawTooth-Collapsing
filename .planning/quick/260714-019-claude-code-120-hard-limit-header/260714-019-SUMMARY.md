# Quick Task 260714-019: Claude Code 上游生命周期与分层超时 Summary

**移除固定 120 秒总 timeout，以 Claude Code context、stream/non-stream 响应头预算、响应 idle 和 hard limit 构成可配置生命周期，并建立协议安全的 504/502 与响应提交边界。**

## Accomplishments

- 默认上游 client 不再设置总 timeout；两套长期复用的 `http.Transport` 分别提供 10m stream 和 30m non-stream 响应头预算，并共享 15s dial、15s TLS 保护。
- hard timeout 从入站 `r.Context()` 派生，Claude Code 取消会直接终止上游且不伪造 502/504；代理自身 header、response-idle、hard timeout 在可提交状态时统一返回 504。
- 非 2xx 与 JSON 在 body 完整读取前不提交状态；SSE 在首个完整事件前不提交 200，提交后的 timeout 只终止流并记录 `response_committed=true`。
- 上游请求显式删除 `Connection` 和复制来的 `Content-Length`，以最终重建 body 设置 `Request.ContentLength`，并以严格单次 RoundTrip 测试证明不重试含糊模型 POST。
- 失败日志记录 `elapsed_ms`、`phase`、`timeout_source`、`stream`，保留普通 URL 诊断信息，同时遮蔽 URL userinfo 且不记录 Authorization、API key 或请求正文。

## Decisions Made

- 保留 Go 默认 Transport 的 proxy、keep-alive、连接池、HTTP/2 协商等行为，只覆盖 dial、TLS 和响应头预算；不按请求创建或 clone Transport。
- 由请求正文的 `stream` 字段在发送前选择 Transport，而不是根据响应 `Content-Type` 事后猜测。
- 所有 response body 读取错误回到同一 classifier；只有代理可明确归责的 header/hard/idle timeout 才是 504，connect/TLS/EOF 不因实现了 `net.Error.Timeout()` 就被泛化为 504。
- 客户端 transport error detail 保持现有契约；敏感边界收紧在 Sawtooth 本地日志，避免扩大外部响应兼容性变化。

**1. [Rule 1 - Compatibility Bug] 为 yaml.v3 裸整数 0 增加 TransportConfig 自定义解码**

- **Found during:** Task 1 GREEN
